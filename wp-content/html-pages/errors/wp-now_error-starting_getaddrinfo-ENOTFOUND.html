What can happen here is that wp-now doesn't know how to route traffic to the website you're trying to set up for the Playground. This is usually because wp-now has already created a Playground instance and doesn't know what to do.

For example, let's say you have already set up a Playground and booted up wp-now for the instance My First Playground

{
	"steps": [
		{
			"step": "defineWpConfigConsts",
			"consts": {
				"WP_HOME": "http://my-first-playground.wpnow:8881",
				"WP_SITEURL": "http://my-first-playground.wpnow:8881"
			},
			"method": "define-before-run"
		}
	]
}

You then want to create an additional instance, My Second Playground, with the following setup:

{
	"steps": [
		{
			"step": "defineWpConfigConsts",
			"consts": {
				"WP_HOME": "http://my-second-playground.wpnow:8881",
				"WP_SITEURL": "http://my-second-playground.wpnow:8881"
			},
			"method": "define-before-run"
		}
	]
}

What will happen is that your machine is going to try to go to the old Playground URL. Because the way wp-now is configured for right now, it's not really easy to work with multiple Playground instances.

On the roadmap we're looking to make wp-now` work better with multiple Playground instances, but for now, it has some challenges when running setup steps - depending on the mode and configuration.

Defining Custom URLs

Defining Custom URLs in wp-now

Here is an example of a Blueprint that defines custom URL constant. wp-now will automatically detect the blueprint and execute it after starting the server. In consequence, the site will be available at http://myurl.wpnow. Make sure myurl.wpnow is added to your hosts file.

{
	"steps": [
		{
			"step": "defineWpConfigConsts",
			"consts": {
				"WP_HOME": "http://myurl.wpnow:8881",
				"WP_SITEURL": "http://myurl.wpnow:8881"
			},
			"method": "define-before-run"
		}
	]
}
